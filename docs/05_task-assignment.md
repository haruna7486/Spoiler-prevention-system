# 役割分担と実装着手順

## 1. 最初に共有する前提

2人が別々に実装しても、最後に接続する場所は同じ `chrome.storage.local` である。

popup側が保存したデータをcontent側が読むため、storageのフィールド名・型・初期値がずれると、各自の画面では動いても結合時に動かない。そのため、UIやDOM操作より先にstorageデータ契約を共有する。

確定済みの契約は [03_architecture.md](./03_architecture.md#4-storageデータ契約) のとおりである。変更提案がある場合は、両担当の合意とドキュメント更新を同じPRに含める。

## 2. 担当一覧

| 担当 | 主担当 | 内容 |
| --- | --- | --- |
| popup / options | はるなさん | React UI、storageの読み書き、キーワード提案UI、視聴済みトグルUI |
| content script | はやとさん | MutationObserver、一致判定、タイトル・サムネイル変更、コメント・おすすめ非表示、動画終了検知 |
| 共通 | 2人 | manifestの結合、共通型、結合テスト、デモ準備 |

MVPではbackground service workerを省略する。したがって、はるなさんの担当はpopup/optionsとstorage操作に集約し、background実装は担当範囲に含めない。

## 3. 実装前の共通作業

2人で最初に次を確認する。

1. [02_tech-stack.md](./02_tech-stack.md) を読み、技術構成を共有する。

5. 共通セットアップPRをmainへ取り込む。
6. 2人のPCで同じcommitをbuildし、Chromeへ読み込めることを確認する。

## 4. はるなさん: popup/options

### 目的

ユーザーがキーワードグループを安全に編集し、その結果を確定済みの形で `chrome.storage.local` へ保存できるようにする。

### 着手順

#### Step 1: storageの読み書きを先に作る

最初に画面デザインを作り込まず、共通の型とstorage操作を接続する。

- 空データを `{ version: 1, groups: [] }` として読める
- グループを保存できる
- Chromeを再起動しても残る
- `enabled` を変更できる
- 保存失敗時にUIへエラーを出せる

共通型はcontent担当も使うため、小さなPRとして先に共有する。

#### Step 2: 最小のグループUIを作る

MVPの順番:

1. グループ一覧
2. グループ追加
3. キーワード追加
4. キーワード削除
5. グループ削除
6. `enabled` トグル
7. 入力エラー表示

見た目の調整より、保存後に再表示して同じ値が出ることを優先する。

#### Step 3: popupとoptionsの役割を整理する

- popup: すぐ使う追加・一覧・トグル
- options: 項目が増えた場合の詳細編集

同じロジックを2画面へコピーせず、hooksやshared componentへ寄せる。MVPでoptionsに置く項目がない場合も、空の画面を増やすのではなく、popupで完結させる範囲をPRに明記する。

#### Step 4: 余力機能

核が結合した後に、次を行う。

- キーワード提案UI
- 視聴済みトグルUI

提案の生成方法や候補データは未確定である。外部APIは使わないという制約を守り、仕様を独断で増やさない。

### 完了条件

- [ ] 追加・削除・再表示ができる
- [ ] `version: 1` と `groups` の形を守っている
- [ ] 空文字と重複キーワードを防げる
- [ ] `enabled` を切り替えられる
- [ ] popupからruntime messageを送っていない
- [ ] `npm run build` が成功する
- [ ] Chrome再起動後もデータが残る

## 5. はやとさん: content script

### 目的

storageの有効キーワードを使い、YouTubeの初期表示・SPA遷移・後読み込みへ追従してネタバレ要素を処理する。

### 着手順

#### Step 1: storage読込と一致判定をDOM操作から分ける

- `version` と `groups` を読む
- `enabled: true` のキーワードだけを取り出す
- 全角半角を正規化する
- 文字列の一致判定を独立した関数として作る
- 正規化と一致判定のテストケースを用意する

最初からYouTubeセレクターと混ぜない。文字列だけで動作を確認できる形にする。

#### Step 2: 静的なDOMで1機能ずつ動かす

推奨順:

1. ホーム画面のタイトル
2. 検索結果のタイトル
3. サムネイル
4. 動画視聴ページの関連動画
5. コメント領域

複数機能を一度に実装すると、セレクター・一致判定・observerのどこが原因か分からなくなる。1機能ごとに対象ページと確認結果をPRへ書く。

#### Step 3: MutationObserverを追加する

初期DOMで処理できることを確認してから、後読み込み対応を追加する。

- observerを重複登録しない
- debounce/throttleを入れる
- 追加された範囲を優先して調べる
- 拡張自身の変更で無限に発火しない
- 同じ要素へ何度適用しても壊れない

#### Step 4: storage変更のリアルタイム反映

`chrome.storage.onChanged` を購読し、変更時に有効キーワードを再構築して表示中DOMを再評価する。

次の両方を確認する。

- キーワード追加後、既存の表示が隠れる
- キーワード削除・無効化後、既存の表示が元へ戻る

#### Step 5: 余力機能

動画終了イベント検知はMVPの核が完成した後に行う。

動画終了だけでは、どのグループを `enabled: false` にするか決められない。動画とグループの対応規則は未確定なので、推測で自動無効化しない。

### 完了条件

- [ ] 初期表示の対象要素を処理できる
- [ ] 後読み込み要素を処理できる
- [ ] storage変更が再読み込みなしで反映される
- [ ] 無効化・削除時に表示を戻せる
- [ ] 同じ要素を再処理しても壊れない
- [ ] YouTube以外では実行されない
- [ ] observerによる目立つ操作遅延がない
- [ ] セレクターが1つ壊れても全処理が停止しない

## 6. つなぎ目

| つなぎ目 | 合意する内容 | 確認方法 |
| --- | --- | --- |
| storage | `version`、`groups`、各フィールドの型 | popup保存後にDevToolsで確認 |
| shared型 | 両担当が同じ定義をimport | 重複型がないことをレビュー |
| 正規化 | 全角半角の変換結果 | 共通テストケース |
| enabled | `false` を判定対象外にする | popup操作後にYouTubeで確認 |
| manifest | popup/options/contentの入口と権限 | `dist/manifest.json` を確認 |
| リアルタイム更新 | `storage.onChanged` を使う | タブ再読込なしで追加・解除 |

### storage契約を最初に合わせる理由

popupとcontentは直接呼び出し合わない。storageが唯一の接続点である。

ここがずれると、たとえばpopupが `keywordGroups` へ保存し、contentが `groups` を読む、といった結合不良が発生する。見た目を作り込んだ後で修正すると、型、UI state、テスト、DOM処理を同時に直すことになる。

最初に小さな共通PRで型・初期値・読込方法を確定すると、2人はその後安全に並行作業できる。

## 7. 結合の順番

1. 共通セットアップとmanifest
2. storage型・初期値・読込/保存
3. popupから手動で1グループを保存
4. contentが保存済みキーワードをconsoleで確認
5. 静的なタイトル1件を伏字化
6. `storage.onChanged` で追加・解除
7. MutationObserverで後読み込み対応
8. サムネイル、コメント、おすすめへ展開
9. 余力機能

最初の結合目標は「popupで1キーワードを追加すると、開いているYouTubeのタイトル1件が再読み込みなしで変わる」である。この縦方向の1本を早く通し、その後に対象画面とUIを広げる。

## 8. メンバーがこの後まずやること

1. リポジトリ管理者がREADMEとdocsを `main` へ最初にpushする。
2. 2人ともリポジトリをcloneし、Node.js 24 LTSとChromeを準備する。
3. 2人で30分の短いキックオフを行い、未確定TODOと初回セットアップ担当を決める。
4. セットアップ担当が `feature/setup` でVite + React + TypeScript + CRXJSの初期構成を作る。
5. 2人のPCで `npm run build` とChromeへの `dist/` 読み込みを確認する。
6. storage型と初期値だけの共通PRを先に作る。
7. はるなさんは `feature/popup`、はやとさんは `feature/content` で並行着手する。
8. 1日目の終わりまでに「popup保存 → content読込」の接続だけを確認する。

UIの完成や全セレクター対応から始めない。最初にビルド、storage契約、最小の縦方向接続を通すことが、手戻りを最も減らす。

- 目的と期待する動作
