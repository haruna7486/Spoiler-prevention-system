# ハッカソン実行・デモガイド

## 1. この文書の目的

この文書は、共有された審査基準、チーム開発の考え方、前回ハッカソンのフィードバックから、このプロジェクトへ直接使える内容だけを実行基準にしたものである。

プロダクト仕様や技術構成は変更しない。評価のために機能を増やすのではなく、価値の核を完成させ、初見の人へ伝えるために使う。

## 2. 評価軸を開発行動へ変換する

| 評価される点 | このプロジェクトで示すこと | 開発中の行動 |
| --- | --- | --- |
| プロダクト価値 | 遅れて視聴する人が、結果を知らずにYouTubeを開ける | 機能ごとではなく利用前後の行動変化で説明する |
| 体験品質・完成度 | 登録から保護・解除までの成功ルートが迷わず動く | 余力機能より、一本のデモ体験を安定させる |
| 技術選定・工夫 | YouTubeのSPAへ追従し、最小権限・サーバーレスで実現する | 技術名ではなく、必要性とトレードオフを記録する |
| 挑戦・学び | 初めてのChrome拡張とチーム開発で、結合リスクを越えた過程 | 詰まり、捨てた案、改善内容をPRへ残す |
| ユーザーへの配慮 | 初見でも登録でき、状態と結果が分かる | 文言、空状態、エラー、即時フィードバックを確認する |
| 発展可能性 | サイト別ルールとversion付きデータ構造 | MVPと将来構想を混同せず説明する |

技術力は加点要素であり、価値と完成度の代替ではない。AIや新しいライブラリを使うこと自体を目的にしない。

## 3. 最小の価値単位

このプロジェクトのMVPは「最小の実装量」ではなく、次の行動変化が起きる最小単位である。

> ユーザーが避けたい対象を登録し、結果を知らずにYouTube上で観たい動画を探せる。

この価値に必要な一本線:

```text
キーワード登録
  ↓
chrome.storage.localへ保存
  ↓
表示中のYouTubeへ即時反映
  ↓
タイトルとサムネイルを保護
  ↓
後読み込み要素にも継続適用
  ↓
無効化すると元へ戻る
```

この一本線が不安定な間は、キーワード提案、視聴済み自動管理、共有機能へ進まない。

## 4. リスク順のマイルストーン

日付ではなく、通過条件で進捗を判断する。

### Gate 1: 2人の環境で同じbuildが動く

- Node.jsと依存バージョンが一致している
- `npm ci` と `npm run build` が成功する
- 2人のChromeで `dist/` を読み込める
- popup、options、content scriptの入口を確認できる

### Gate 2: storage契約がつながる

- popupが確定済みデータ形式で保存する
- contentが同じ型と初期値で読み込む
- `enabled` の意味が両担当で一致している

### Gate 3: 最小の縦方向接続が動く

- popupで1キーワードを追加できる
- 表示中のタイトル1件が再読み込みなしで変わる
- 無効化または削除で元に戻る

### Gate 4: YouTubeの実利用へ広げる

- ホーム、検索、動画視聴ページの対象を確認する
- サムネイル、関連動画、コメントへ展開する
- MutationObserverで後読み込みを処理する
- 重複処理や目立つ性能劣化がない

### Gate 5: デモを固定する

- 成功ルートを複数回連続で再現できる
- 2台目のPCでも動く
- build、拡張再読み込み、テスト用データの手順が固定されている
- デモ動画とスクリーンショットを用意する

未知の技術、初回結合、デモ依存部分を後半まで残さない。

## 5. UXの完了基準

popupが「動く」だけで完了にしない。初めて使う人が、説明なしで現在の状態と次の操作を理解できることを目指す。

### 入力

- 空文字を登録できない
- 重複キーワードを登録できない
- エラー理由を入力欄の近くへ表示する
- ボタンの文言から実行内容が分かる

### フィードバック

- 保存できたか分かる
- 保存中に同じ操作を連打しても壊れない
- 失敗をconsoleだけで終わらせず、ユーザーへ伝える
- `enabled` の現在状態が見た目と文言で分かる

### 空状態と初回利用

- グループが0件でも壊れた画面に見えない
- 最初のキーワードを追加する行動が分かる
- 専門用語を避け、必要な場合は短い説明を付ける

### YouTube上の体験

- 何が拡張によって隠されたか判別できる
- 誤検知時にユーザーが解除できる
- 無効化・削除すると表示が戻る
- ネタバレを隠す表示自体が動画選択を過度に妨げない

色、文言、サムネイル差し替えデザインは未確定事項である。実装前に独断で固定せず、最初の試作品を2人で触って決める。

## 6. 最小テスト方針

短期間でも、デモの核が壊れたことを早く検知できる確認は残す。

### 自動化を優先するロジック

- 全角半角の正規化
- キーワード一致・不一致
- `enabled: false` の除外
- 空・重複キーワードの検証
- storageの初期値とversion判定

テストランナーは未確定である。導入する場合は技術追加の目的とコストをPRに書く。導入前も、入力と期待結果の一覧を残して手動確認できるようにする。

### 実ブラウザで確認する体験

- 初期表示への適用
- storage変更の即時反映
- SPA遷移と後読み込み
- 同じ要素を複数回処理したときの冪等性
- 無効化・削除後の復元
- YouTube以外で実行されないこと

PR前は `npm run build` に加え、変更した価値の流れをChromeで確認する。

## 7. チーム運用

### キックオフ

最初に次を仮決めする。

- チームとして狙うこと
- 各自が得たい経験
- 作業可能時間と同期できる時間
- 初回セットアップ担当
- 詰まりを共有するまでの目安
- 意見が割れたときの決め方

### コミュニケーション

- 設計、詰まり、認識合わせは同期して話す
- 決定、進捗、後から参照する内容はテキストへ残す
- 反対意見と初歩的な質問を歓迎する
- 担当外でも結合点には相互に関与する

### 短い共有で話す順番

1. 一番危ないこと
2. 詰まっていること
3. 助けが必要なこと
4. 次に通す最小の確認

完了報告より、放置すると価値の核を止める問題を先に共有する。

## 8. 意思決定と学びの残し方

仕様を変更・削除したときは、IssueまたはPRへ次を残す。

```text
決定:
理由:
採用しなかった案:
確認できたこと:
まだ確認できていないこと:
次に再検討する条件:
```

発表前にログを読み返し、次を自分たちの言葉へまとめる。

- 最も難しかった点
- 最初の想定が外れた点
- 2人で判断を変えた理由
- Chrome拡張・React・チーム開発で学んだこと
- 時間があれば次に改善すること

成功だけでなく、試行錯誤と判断の変化が挑戦の証拠になる。

## 9. デモ成功ルート

### 見せる体験

1. 事前に用意したYouTube画面で、対象のタイトルとサムネイルを見せる。
2. popupを開き、作品・イベント名とキーワードを登録する。
3. 同じタブでタイトルとサムネイルが即時に保護されることを見せる。
4. スクロールし、後から表示された対象も保護されることを見せる。
5. `enabled` をOFFにし、表示が戻ることを見せる。

コメント・おすすめ非表示まで安定している場合だけ成功ルートへ加える。実装済みでも不安定な機能は、デモの主経路から外す。

### デモデータ

- 一目で一致が分かるキーワードを使う
- 本番のネタバレを観客へ見せない題材を選ぶ
- ログイン状態やおすすめ内容に依存しすぎない
- YouTube側の表示変更に備えて、直前に対象ページを確認する

### バックアップ

- 成功ルートの操作動画
- Before / Afterの連続スクリーンショット
- PDF化した発表資料
- 同じcommitを動かせる別メンバーのPC
- デモ復旧手順

この拡張はYouTubeという外部サイトのDOMと通信環境に依存する。録画とスクリーンショットは任意ではなく、デモ設計の一部とする。

## 10. 発表の最小構成

### 最初の30秒

次を即答する。

- 誰の: リアルタイムで観られず、後から動画を観る人
- 何が: YouTubeを開く前後にタイトルや画像で結果を知ってしまう
- どう変わる: 避けたい対象を登録し、結果を知らずに観たい動画へたどり着ける
- 何が違う: タイトルだけでなく画像と後読み込み要素もまとめて保護する

### 発表全体

1. 課題と具体的な利用場面
2. 既存手段では残る問題
3. デモによるBefore / After
4. 価値を成立させる技術的な工夫
5. 挑戦、失敗、判断変更、学び
6. 現在の限界と今後

専門用語を並べず、技術は「なぜその体験に必要だったか」から説明する。

## 11. Q&Aで準備する問い

- なぜキーワードをユーザーが登録する方式なのか
- なぜAIでネタバレ判定しないのか
- 既存のネタバレブロッカーと何が違うのか
- 誤検知・見逃しをどう扱うのか
- YouTubeのDOM変更へどう備えるのか
- なぜbackgroundやサーバーが不要なのか
- ユーザーの閲覧情報は外部へ送られるのか
- YouTube以外へどう拡張できるのか
- 2人が最も苦労し、学んだことは何か

答えは「結論 → 理由 → 現在確認できている範囲 → 次に行うこと」の順にする。未検証のことを実現済みのように答えない。

## 12. 発表前チェック

### 価値

- [ ] 誰のどの場面を変えるか、30秒で言える
- [ ] 機能ではなくBefore / Afterで説明できる
- [ ] 既存手段との差を誇張せず説明できる

### 体験

- [ ] 登録から保護・解除まで迷わず操作できる
- [ ] エラー、空状態、保存結果がユーザーに分かる
- [ ] デモ成功ルートを連続して再現できる

### 技術

- [ ] 技術選定の理由とトレードオフを説明できる
- [ ] `main` のbuildが成功する
- [ ] 不要な権限や外部通信がない

### 挑戦・学び

- [ ] 2人それぞれが自分の実装を説明できる
- [ ] 詰まりと解決過程を自分たちの言葉で話せる
- [ ] 捨てた機能と判断理由を説明できる

### バックアップ

- [ ] 動画、スクリーンショット、PDFがある
- [ ] 別PCでも同じcommitが動く
- [ ] デモが失敗した場合の切り替え方を練習した
